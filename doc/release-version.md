# ZrLog 版本发布与 Changelog 编写规范

为了在未来的版本迭代中保持发布日志的统一性、专业性以及可读性，特制定本规范指南。建议在每次发布新版本（Release）时参考此文件。

## 1. 变更日志 (Changelog) 格式规范

发布日志采用标准的 Markdown 格式，层级分明，主要分为 **标题**、**新特**、**优化**、**修复**、**依赖** 五大板块。

### 1.1 标准模板

```markdown
### x.x.x (YYYY-MM-DD)

#### 新特

- 【模块名称】新增特性的简要描述（如涉及特定条件，可在括号中补充说明）
- 【模块名称】新增特性的简要描述

#### 优化

- 【模块名称】功能优化、逻辑重构或性能提升的描述
- 【模块名称】UI体验改善、构建流程优化的描述

#### 修复

- 【模块名称】修复了什么具体问题，解决了什么报错或异常
- 【模块名称】修复漏洞或不兼容问题

#### 依赖

- ⬆️ 依赖包/组件名称 [新版本号]
- ⬆️ 依赖包/组件名称 [新版本号] （补充说明，如：仅构建）
- ⬇️ 依赖包/组件名称 [降级版本号]
```

### 1.2 模块名称前缀参考字典

为了保持日志排版的整齐划一，所有条目前必须附带由 `【】` 包裹的模块标识。推荐使用以下标准模块名：

- **【底层】**：对应 `zrlog-base` 或 `zrlog` 核心模块的变动。
- **【管理页】**：对应 `zrlog-admin-web` 的后台管理界面和逻辑变动。
- **【博客页】**：对应 `zrlog-blog-web` 或前端主题相关的变动。
- **【编辑器】**：对应 `zrlog-editor` Markdown 相关的组件更新。
- **【安装页】**：对应 `zrlog-install-web` 系统安装初始化向导的变动。
- **【插件】**：对应各周边插件（如 `zrlog-plugin-cos`, `zrlog-plugin-static-plus` 等）。
- **【构建】**：对应打包发布、CI/CD流水线、Maven/Docker 相关配置的变动。
- **【部署】**：对应 Nginx、Docker 运行环境脚本的优化或变动。
- **【官网】**：对应 `zrlog-www` 主站的更新。
- **【开发】**：面向开发者的基础变动（如接口规范调整、全局错误处理等）。

## 2. Git Commit 提交规范建议

为了在发布新版时能够更快速、精准地从 `git log` 中提炼变更日志，建议在日常开发中使用语义化提交（Semantic Commit Messages）规范：

- `feat:` 新功能（会被归类到 **新特**）
    - *例: feat(管理页): 支持设置友情链接和导航的图标*
- `fix:` 修复 Bug（会被归类到 **修复**）
    - *例: fix(编辑器): 修复在特定环境下文件上传失败的问题*
- `perf:` 性能优化或体验改善（会被归类到 **优化**）
    - *例: perf(管理页): 缓存更新更及时*
- `refactor:` 代码重构（会被归类到 **优化**）
- `chore:` 构建、依赖包更新或辅助工具的变动（会被归类到 **优化** 或 **依赖**）
    - *例: chore(构建): 升级 Tomcat 至 10.1.49*
- `docs:` 文档变更

## 3. 版本发布检查清单 (Release Checklist)

在打 Tag 并编写发布日志前，推荐通过以下清单进行 Review：

1. **提取变更日志**：
    - 运行 `git log <上一版本Tag>..HEAD --oneline` 提取所有相关仓库的提交记录。
2. **剔除无用信息**：
    - 过滤掉内部开发相关的杂乱提交（如 `Fix upload error` 这种重复提交合并为一条）。
    - 过滤掉 `shell-release` 等版本号 Bump 的自动化提交。
3. **分类与转译**：
    - 将技术视角的 Commit（如 `get status code`）转换为面向用户的表达（如 `【底层】重构并完善处理 HTTP 状态码的内部逻辑`）。
4. **依赖梳理**：
    - 检查根目录 `pom.xml` 或核心组件的 `package.json`，罗列出发生了大版本跨越或关键升级的依赖项。
5. **排版格式化**：
    - 严格按照第 1 节的标准模板进行 Markdown 格式化并发布。
